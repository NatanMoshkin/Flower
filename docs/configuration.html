System architecture

Single-panel PC hosts the whole stack. The TwinCAT runtime drives 6 SOL-Spring pneumatic pistons (single Extend coil, mechanical spring returns them) over EtherCAT; the TcHmi runtime serves the web HMI over WebSocket; and a small Python bridge (RobotBridge/robot_bridge.py) does bidirectional TCP/IP with the robot — receiving position frames and forwarding master-cycle events — translating each side into ADS reads/writes on the PLC.

Key design choice. The PLC has zero networking code. The whole TCP/IP layer lives in RobotBridge/ as a single-file Python service. This deliberately sidesteps Beckhoff's paid TF6310 TCP/IP supplement.
Direction of the socket. The robot is the TCP server, the bridge is the client and dials in at 192.168.201.1:6001. This matches the reference client in RobotBridge/Client_working_example/. Older versions of these docs described the opposite — that arrangement is gone.

Function block — FB_SolSpringPiston_2Pos

One FB drives every piston instance (six today). SOL-Spring topology: a single Extend coil, mechanical spring returns the piston when the coil is de-energised. Two limit switches (Extended and Retracted) still exist on the DI side. Sections in scan order:

  • 0a — Effective sensor merge: bSwitchExtended_eff := bSwitchExtended OR stHmi.bSimulateExtended (same for retracted).
  • 0 — HMI button latching: R_TRIG on bManCmdExtend / bManCmdRetract selects the latched target. Independent IFs — retract wins on same-cycle collision.
  • 1 — Mode → target mapping: Auto uses bAutoCmd* pins, Manual uses bSelected* latches, otherwise safe-default zeros.
  • 1a — Jog override: while either bManJog* button is held, latched selections are cleared.
  • 1b — Target-reached reset: once the piston arrives at a sensor, the corresponding latched selection is dropped.
  • 2 — Coil drive ladder (latch & retain): Jog Extend energises the coil while held (Manual-mode gated); Jog Retract de-energises. Otherwise the coil is latched — once TRUE by an Extend target it stays TRUE until an explicit Retract target flips it. No ELSE branch: the mechanical spring, not the ladder, is what retracts the piston, so the coil must not drop simply because the target has been reached.
  • 3 — HMI status feedback: mirrors effective sensor + coil state into stHmi. bIsMoving := NOT bSwitchExtended_eff AND NOT bSwitchRetracted_eff.
Latch semantics matter here. A SOL-SOL piston keeps position after both coils de-energise. A SOL-Spring piston does not — if the coil drops mid-cycle the spring pulls the actuator back. The Section 2 ladder is written specifically to hold the coil high until the next command; do not "clean it up" by adding an ELSE branch that clears the coil at the target.

Function block — FB_MasterAutoCycle

Coordinates 1..3 pairs of (SepSolN, PushSolN) pistons in parallel with the robot's Pos1/Pos2 signals. Ten-state machine. See the CLAUDE.md DECISION section for the full state table; summary:

  • IDLE. Wait for operator START or stRobot.bAtPos1 rising or bSimStep_Idle. On trigger: emit AUTO_STARTED to the robot.
  • SEP_EXTENDING → WAIT_POS2 → PUSH_EXTENDING → DWELL_PUSH → PUSH_RETRACTING (emits PUSH_DONE on sensor-reached) → PUSH_RETRACTED_DWELL → SEP_RETRACTING → SEP_RETRACTED_DWELL → IDLE.
  • ERR. Latched. Emits PISTONS_ERROR. iErrorCode = step's fault code (1=SEP_EXTENDING timeout, 2=WAIT_POS2, 3=PUSH_EXTENDING, 4=PUSH_RETRACTING, 5=SEP_RETRACTING, 99=STOP-while-running). Reset on HMI Reset rising edge or stRobot.bRxResetError rising edge.

Coexists with the per-piston FB_PistonAutoCycle tiles via OR-merge on bAutoCmd* at each piston call site. While the master is running (non-IDLE, non-ERR), MAIN forces the enabled pistons into Automatic mode so the master's commands actually reach the coils; when it's IDLE or ERR, MAIN preserves the operator's manual choice.

MAIN scan structure

Scan order in PLC1/POUs/MAIN.TcPOU:

  1. READ INPUTS. Copy GVL_IO.dIn[N] to GVL_App.bSensor*. Wired for SepSol1 (dIn[1..2]) and PushSol1 (dIn[7..8]); SepSol2/3 + PushSol2/3 lines are commented placeholders (dIn[??]) pending channel assignment.
  2. Call fbMasterAutoCycle. Reads sensor bIs* from all six piston structs, reads Pos1/Pos2 from stRobot, writes Tx pulses back into stRobot. Called first so its bSepCmd*[N] / bPushCmd*[N] outputs feed forward this scan.
  3. Mode-forcing block. When fbMasterAutoCycle.bAnyRunning is TRUE, forces stSep/PushSolN.eMode := Automatic for indices ≤ uiPairCount. Otherwise no-op (operator's manual choice is preserved).
  4. Call fbSepSol1Auto and fbPushSol1Auto. Per-piston tiles for bench-testing loops on one piston.
  5. Call six piston FBs. fbSepSol1..3, fbPushSol1..3. Each bAutoCmd* is an OR-merge of the corresponding per-piston auto FB (where present) and the master's array output — e.g. fbSepSol1Auto.bCmdExtend OR fbMasterAutoCycle.bSepCmdExtend[1].
  6. WRITE OUTPUTS. Copy GVL_App.bValve* to GVL_IO.dOut[N]. Layout is fully assigned on DO: dOut[1..3] = SepSol1..3 Extend coils, dOut[4..6] = PushSol1..3 Extend coils, dOut[7..8] are freed (previously PushSol1's two SOL-SOL coils; now unused, kept assigned to FALSE).
Soft-only pistons. The DO side is fully assigned. The DI side is still partial — SepSol2/3 and PushSol2/3 have no sensor channels wired, so their "Extended" / "Retracted" indicators light up only via the operator's Simulate checkboxes. See Open issues for the channel table.

Naming convention

Follows the project's TwinCAT_NamingConventions skill (Structured Hungarian):

  • b* BOOL, n* integer (UDINT/DINT/INT), r* REAL/LREAL, s* STRING, e* ENUM, st* STRUCT instance, fb* FB instance.
  • FB_* function block declaration, ST_* struct type, E_* enum type, GVL_* global variable list.
  • Signals prefixed by role: bSensor* (inputs), bValve* (outputs), bMan* (HMI-writes), bSelected*/bIs*/bOut* (FB-writes).

Two-schema model — where each symbol is defined

TwinCAT HMI uses two schema files that must agree for a symbol to be visible AND writable:

FileRole
Properties/tchmi.project.Schema.jsonProject schema. Type definitions used by the designer / IntelliSense. Contains PLC1.E_ControlMode, PLC1.E_RobotConnState, PLC1.ST_HmiPiston, PLC1.ST_HmiRobot.
Server/TcHmiSrv/TcHmiSrv.Config.default.jsonServer schema. ADS-side type definitions (ADS-PLC1.*), SYMBOLS map (ADS.PLC1.GVL_HMI.stSepSol1, ..., ADS.PLC1.GVL_Robot.stRobot), and USERGROUPS ACL grants (each symbol must be listed in USERGROUPS.__SystemUsers.SYMBOLS with access value 3 = R/W).
Gotcha. Forgetting the USERGROUPS.__SystemUsers.SYMBOLS grant is the #1 cause of SYMBOL_NOT_AUTHORIZED errors that survive "Activate Configuration". Adding the SYMBOLS map entry alone isn't enough.

Robot Bridge — Python + pyads

Location: RobotBridge/ at the repo root. Single-file Python service, no framework. Runs as a long-lived process (foreground console, Scheduled Task, or NSSM service).

sequenceDiagram
  participant Robot as Robot (server, :6001)
  participant Bridge as robot_bridge.py (client)
  participant PLC as PLC (GVL_Robot.stRobot)
  participant HMI as HMI (Robot & AutoCycle pages)

  Note over Bridge: startup — load config.yaml, open ADS
  Bridge->>PLC: eConnState = Connecting
  Bridge->>Robot: TCP connect 192.168.201.1:6001
  Robot-->>Bridge: SYN-ACK, socket up
  Bridge->>PLC: eConnState = Connected

  loop main loop (2 s socket timeout)
    alt inbound frame arrives
      Robot->>Bridge: "POS2\n"
      Bridge->>PLC: nPacketsRx++, sLastMessage="POS2", bAtPos2=T
      PLC-->>HMI: (subscription) POS2 indicator lights
    else RESET_ERROR frame
      Robot->>Bridge: "RESET_ERROR\n"
      Bridge->>PLC: bRxResetError = TRUE
      Note over PLC: FB_MasterAutoCycle rising-edge
consumes flag, exits ERR
    else recv times out (no data)
      Bridge->>Bridge: idle for socket_timeout
    end

    Bridge->>PLC: read bTxAutoStarted / bTxPushDone / bTxPistonsError
    alt any Tx flag = TRUE
      Bridge->>Robot: "AUTO_STARTED\n" (or PUSH_DONE / PISTONS_ERROR)
      Bridge->>PLC: nPacketsTx++, sLastTxMessage="AUTO_STARTED", clear the flag
    end

    opt idle > heartbeat_idle_seconds
      Bridge->>Robot: "HEARTBEAT" bytes (drop-detector, no reply expected)
    end
  end

  Robot--xBridge: connection dropped
  Bridge->>PLC: eConnState = Disconnected
  Note over Bridge: sleep reconnect.delay_seconds, redial

Wire format

ASCII, line-terminated by \n. Case-insensitive. Trailing \r is stripped. Frame tokens are commissioning-tunable — the constants live in PlcRobotSymbol.TX_FRAMES and the parser in parse_position / is_reset_error.

Robot → Bridge → PLC (inbound):

FrameEffect on GVL_Robot.stRobot
POS1\nbAtPos1 = TRUE, bAtPos2 = FALSE, bAtPos3 = FALSE. nPacketsRx++, sLastMessage="POS1".
POS2\nSame, but Pos2 latched. FB_MasterAutoCycle's WAIT_POS2 state exits on Pos2 rising.
POS3\nSame, but Pos3 latched.
RESET_ERROR\nbRxResetError = TRUE (one-scan pulse — FB consumes on rising edge, clears the flag). Master cycle exits ERR back to IDLE.
anything elsenPacketsRx++, sLastMessage updates, but no state change. Useful for surfacing malformed frames on the HMI.

PLC → Bridge → Robot (outbound Tx pulses, drained once per main-loop iteration):

PLC flagFrame sentTrigger
bTxAutoStartedAUTO_STARTED\nMaster cycle leaves IDLE (operator START or Pos1 rising).
bTxPushDonePUSH_DONE\nMaster cycle completes PUSH_RETRACTING (all enabled push pistons at Retracted).
bTxPistonsErrorPISTONS_ERROR\nMaster cycle enters ERR (iErrorCode set to the step's fault code).
No queue, no lost pulses. If the bridge is disconnected when the PLC pulses a Tx flag, the flag stays TRUE. On next reconnect the bridge's poll picks it up, sends the frame, and clears it. The operator's view (nPacketsTx, sLastTxMessage) advances only on actual send.
"In position" semantics. Position bits are latched: bAtPos2 stays TRUE until the robot sends a different POSn frame. The bridge does not clear on timeout.

Failure handling

  • Robot refuses connection: bridge catches the exception, sets eConnState = Error, sleeps reconnect.delay_seconds, retries.
  • Robot unreachable / socket timeout on connect: same recovery path.
  • Connection drops mid-session: recv() returns b"" or the socket raises. Bridge closes, sets eConnState = Disconnected, sleeps, redials. SO_KEEPALIVE is enabled so a half-open TCP (peer power loss, cable pulled) is eventually detected by the OS.
  • Half-open TCP faster than OS keepalive: the idle-probe heartbeat catches it. After heartbeat_idle_seconds with no inbound data, the bridge tries to send heartbeat_payload. If send() raises, the recovery path fires; no reply is required.
  • ADS write fails: logged as exception, top-level supervisor catches it, sets eConnState = Error, sleeps, retries.
  • Bridge process itself dies: HMI keeps showing the last-written eConnState — no PLC-side watchdog in v1.

Licensing rationale

Beckhoff's native Tc2_TcpIp library requires the paid TF6310 supplement + install on the runtime target. Same for Tc3_IoT_Communicator (TF6701 MQTT) and OPC UA server (TF6100). Deliberately avoided in this project.

ADS is built into TwinCAT (both runtime and client DLL are free), and pyads is MIT-licensed. The bridge is therefore royalty-free on both sides. See the memory note feedback_avoid_licensed_beckhoff_libs for the durable version of this decision.
