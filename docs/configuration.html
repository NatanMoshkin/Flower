System architecture

Single-panel PC hosts the whole stack. The TwinCAT runtime drives 6 SOL-SOL pneumatic pistons over EtherCAT; the TcHmi runtime serves the web HMI over WebSocket; and a small Python bridge (RobotBridge/robot_bridge.py) translates the robot's ASCII TCP/IP frames into ADS writes on the PLC.

Key design choice. The PLC has zero networking code. The whole TCP/IP layer lives in RobotBridge/ as ~230 lines of Python. This deliberately sidesteps Beckhoff's paid TF6310 TCP/IP supplement.

Function block — FB_SolSolPiston_2Pos

One FB drives every piston instance (six today). SOL-SOL topology with two limit switches. Sections in scan order:

  • 0a — Effective sensor merge: bSwitchExtended_eff := bSwitchExtended OR stHmi.bSimulateExtended (same for retracted).
  • 0 — HMI button latching: R_TRIG on bManCmdExtend / bManCmdRetract selects the latched target. Independent IFs — retract wins on same-cycle collision.
  • 1 — Mode → target mapping: Auto uses bAutoCmd* pins, Manual uses bSelected* latches, otherwise safe-default zeros.
  • 1a — Jog override: while either bManJog* button is held, latched selections are cleared.
  • 1b — Target-reached reset: once the piston arrives at a sensor, the corresponding latched selection is dropped.
  • 2 — Solenoid drive ladder: Jog branches ahead of latched targets (Manual-mode gated). Effective sensor stops the coil. Same-cycle collision: retract wins.
  • 3 — HMI status feedback: mirrors effective sensor + coil state into stHmi.

GVLs (global variable lists)

GVLRole
GVL_IOThe only hardware-mapped symbols: dIn AT %I* : ARRAY[1..16] OF BOOL, dOut AT %Q* : ARRAY[1..16] OF BOOL. All EtherCAT links terminate here.
GVL_AppPlain-BOOL aliases with descriptive names (bSensorSepRetracted1, bValveSepExtend1, ...). MAIN copies raw GVL_IO.dIn[N] into these at the top of the scan, and the reverse at the bottom.
GVL_HMISix ST_HmiPiston instances (stSepSol1/2/3, stPushSol1/2/3). {attribute 'TcHmiSymbol.AddSymbol'} pragma auto-registers them with the HMI symbol table.
GVL_RobotOne ST_HmiRobot instance stRobot. Also auto-registered via the same pragma. Written by the Python bridge over ADS, read by the HMI Robot page.

MAIN scan structure

Scan order in PLC1/POUs/MAIN.TcPOU:

  1. READ INPUTS. Copy GVL_IO.dIn[N] to GVL_App.bSensor*. Currently wired for SepSol1 (dIn[1..2]) and PushSol1 (dIn[7..8]); SepSol2/3 + PushSol2/3 lines are commented with TODO markers pending channel assignment.
  2. Call six piston FBs. fbSepSol1..3, fbPushSol1..3, each with bAutoCmd* pins wired to FALSE (future FB_Auto hook).
  3. WRITE OUTPUTS. Copy GVL_App.bValve* to GVL_IO.dOut[N]. Same partial wiring — SepSol1 (dOut[1..2]) and PushSol1 (dOut[7..8]) only.
Soft-only pistons. SepSol2/3 and PushSol2/3 currently run software-only: the HMI operator can drive them and see feedback, but the coil bits never reach hardware. See Open issues for the channel-assignment table.

Naming convention

Follows the project's TwinCAT_NamingConventions skill (Structured Hungarian):

  • b* BOOL, n* integer (UDINT/DINT/INT), r* REAL/LREAL, s* STRING, e* ENUM, st* STRUCT instance, fb* FB instance.
  • FB_* function block declaration, ST_* struct type, E_* enum type, GVL_* global variable list.
  • Signals prefixed by role: bSensor* (inputs), bValve* (outputs), bMan* (HMI-writes), bSelected*/bIs*/bOut* (FB-writes).

Two-schema model — where each symbol is defined

TwinCAT HMI uses two schema files that must agree for a symbol to be visible AND writable:

FileRole
Properties/tchmi.project.Schema.jsonProject schema. Type definitions used by the designer / IntelliSense. Contains PLC1.E_ControlMode, PLC1.E_RobotConnState, PLC1.ST_HmiPiston, PLC1.ST_HmiRobot.
Server/TcHmiSrv/TcHmiSrv.Config.default.jsonServer schema. ADS-side type definitions (ADS-PLC1.*), SYMBOLS map (ADS.PLC1.GVL_HMI.stSepSol1, ..., ADS.PLC1.GVL_Robot.stRobot), and USERGROUPS ACL grants (each symbol must be listed in USERGROUPS.__SystemUsers.SYMBOLS with access value 3 = R/W).
Gotcha. Forgetting the USERGROUPS.__SystemUsers.SYMBOLS grant is the #1 cause of SYMBOL_NOT_AUTHORIZED errors that survive "Activate Configuration". Adding the SYMBOLS map entry alone isn't enough.

Localization

Localization/en.localization + de.localization. Current keys: L_StartPage, L_Piston, L_Robot, L_Events, L_Localization, L_Theme, L_Settings. Both files must be kept in sync when adding a new key.

Robot Bridge — Python + pyads

Location: RobotBridge/ at the repo root. Single-file Python service, no framework. Runs as a long-lived process (foreground console, Scheduled Task, or NSSM service).

Wire format

ASCII, line-terminated by \n. Case-insensitive. Trailing \r is stripped.

FrameEffect on GVL_Robot.stRobot
POS1\nbAtPos1 = TRUE, bAtPos2 = FALSE, bAtPos3 = FALSE. Counter and last-message updated.
POS2\nSame, but Pos2 latched.
POS3\nSame, but Pos3 latched.
anything elseCounter increments, sLastMessage updates, but no bAtPos* change. Useful for surfacing malformed frames on the HMI.
"In position" semantics. Position bits are latched: bAtPos2 stays TRUE until the robot sends a different POSn frame. The bridge does not clear on timeout in v1.

Failure handling

  • Robot never connects (server mode): bridge blocks on accept(). eConnState = Connecting stays until a connection arrives.
  • Robot refuses connection (client mode): bridge catches ConnectionRefusedError, sets eConnState = Error, sleeps reconnect.delay_seconds, retries.
  • Connection drops mid-session: recv() returns b"", bridge closes the socket, sets eConnState = Disconnected, sleeps, reconnects.
  • ADS write fails: logged as exception, top-level supervisor catches it, sets eConnState = Error, sleeps, retries.
  • Bridge process itself dies: HMI keeps showing the last-written eConnState — no PLC-side watchdog in v1. Future FB_RobotWatchdog would reset the state based on nPacketsRx stagnation.

Licensing rationale

Beckhoff's native Tc2_TcpIp library requires the paid TF6310 supplement + install on the runtime target. Same for Tc3_IoT_Communicator (TF6701 MQTT) and OPC UA server (TF6100). Deliberately avoided in this project.

ADS is built into TwinCAT (both runtime and client DLL are free), and pyads is MIT-licensed. The bridge is therefore royalty-free on both sides. See the memory note feedback_avoid_licensed_beckhoff_libs for the durable version of this decision.
