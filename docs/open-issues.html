#1 — Robot TCP/IP role: server or client? Deferred

Owner: commissioning · Blocks: real-robot integration only · Introduced: initial robot bridge commit

Question. Does the robot dial the bridge (bridge = TCP server), or does the bridge dial the robot (bridge = TCP client)?

Why deferred. Different robot vendors default to different roles: some expose their own TCP server (bridge should be client), others expect a listening endpoint (bridge should be server). Choosing before we know the specific controller model risks committing code that has to be flipped anyway.

How to resolve. One line in RobotBridge/config.yaml:

robot:
  role: "server"   # or "client"
  host: "192.168.1.50"   # ignored when role=server
  port: 2000

Restart the bridge process. No code change, no rebuild.

Default in git. role: "server", port: 2000, placeholder host. This works out of the box for the nc-based smoke test.

Related code. RobotBridge/robot_bridge.py: accept_from_robot() vs connect_to_robot() — both feed the same parser and PLC writer, so the choice is truly cosmetic in the source.

Broader constraint. This choice sits inside a bigger decision — the whole bridge exists to avoid Beckhoff's paid TF6310 TCP/IP supplement. Do not "solve" this by adding Tc2_TcpIp to the PLC project; see configuration.html#bridge and the memory note feedback_avoid_licensed_beckhoff_libs for the rationale.

#2 — Physical IO channel assignment for SepSol2/3 + PushSol2/3 Blocking hw

Owner: electrical integrator · Blocks: full-rig operation · Software already wired

Question. Which GVL_IO.dIn[] / dOut[] array slots are wired to the physical proximity switches and solenoid valves of the four unassigned pistons?

Current state. GVL_IO.dIn and GVL_IO.dOut are sized ARRAY[1..16]. Slots 1..8 were re-verified after the array grew; slots 9..16 are unassigned. Two pistons are wired:

ActuatorDI (Retract)DI (Extend)DO (Retract)DO (Extend)
SepSol1dIn[1]dIn[2]dOut[1]dOut[2]
SepSol2TBDTBDTBDTBD
SepSol3TBDTBDTBDTBD
PushSol1dIn[7]dIn[8]dOut[7]dOut[8]
PushSol2TBDTBDTBDTBD
PushSol3TBDTBDTBDTBD

What "soft-only" means. The four unassigned pistons already run — HMI operator can drive them and see UI feedback — but the FB outputs never reach hardware. The GVL_App alias variables stay in software; the copies to/from GVL_IO in MAIN's READ INPUTS / WRITE OUTPUTS bookends are commented out with TODO markers.

How to resolve. Once channels are assigned:

  1. Fill in the four TBD rows above.
  2. Open Flower_PLC_HMI/PLC1/POUs/MAIN.TcPOU. Ten lines are currently commented placeholders like // GVL_App.bSensorSepRetracted2 := GVL_IO.dIn[??];.
  3. Replace each [??] with the actual slot number, uncomment the line, and remove its TODO marker.
  4. Rebuild + Activate. In the Watch window, force GVL_IO.dIn[N] for each new switch and confirm the corresponding bAtPos / sensor indicator on the HMI updates.

Tag convention. <Device>Sol<N><slot> where slot 1 = RETRACT, slot 2 = EXTEND. E.g. the SepSol1 retract sensor is DiSepSol11 and the extend sensor is DiSepSol12.

Related caveat. After the array grew from [1..8] to [1..16], the EtherCAT links for slots 1..8 must be re-verified in the System Manager — TwinCAT does not always re-attach linked variables when their parent array is resized. Symptom: SepSol1 / PushSol1 sensor indicators stop updating even though nothing else changed.
