Install

First-time setup on a fresh panel PC. Everything is local by default — the docs, HMI, PLC runtime, and Python bridge all target 127.0.0.1. Only the robot lives at a real IP.

Prerequisites

  • TwinCAT 3.1 (build 4026 or newer) with the TcXaeShell IDE.
  • TE2000 TwinCAT HMI Engineering — free for engineering use.
  • Python 3.9+ on PATH. Verify with python --version.
  • Git. Clone with git clone https://github.com/NatanMoshkin/Flower.git.
No paid Beckhoff supplements are required. Notably, TF6310 TCP/IP is not installed and must not be referenced from PLC1 — the robot channel is handled by the Python bridge instead.

1. PLC — build & activate

  1. Open Flower_PLC_HMI/Flower_PLC_HMI.sln in TcXaeShell.
  2. Build → Build Solution. Zero errors expected.
  3. TwinCAT → Activate Configuration. Restart the runtime in Run mode when prompted.
  4. Verify: PLC → Login, then in the online view confirm GVL_Robot.stRobot is present and stRobot.Name = "Robot".

2. HMI — build & publish

  1. Right-click the HMI project → Build.
  2. Right-click → Publish → Local Engineering Server. Note the URL (usually http://127.0.0.1:1010/ for engineering, http://127.0.0.1:3003/ for the runtime WebSocket).
  3. Open the browser at the engineering URL. Log in as __SystemUser (passwordless in the default config).
  4. Confirm the navigation menu shows: StartPage, Piston, Robot, Events, Settings.

3. Python bridge — install dependencies

cd RobotBridge
pip install -r requirements.txt

Installs pyads and pyyaml. On Windows the ADS client DLL ships with TwinCAT — no extra install. On Linux, install libads per the pyads setup docs.

4. Python bridge — create config.yaml

cd RobotBridge
cp config.example.yaml config.yaml
# edit config.yaml — set robot.host + robot.port + robot.role

Defaults (server mode, port 2000, placeholder IP) are enough to run the smoke test with nc. Real robot integration requires editing host / port to match the robot controller and picking a role.

5. ADS route (remote hosts only)

Skip this step if the bridge runs on the same PC as TwinCAT — the local route 127.0.0.1.1.1 is present out of the box.

If the bridge runs on a different machine:

  1. On the TwinCAT engineering PC: System Manager → Routes → Add Route.
  2. Point at the bridge host's AMS Net ID (usually <host-ip>.1.1).
  3. Update plc.ams_net_id in RobotBridge/config.yaml accordingly.

6. Docs (this site)

Double-click scripts\serve-docs.bat. Serves the repo root on http://127.0.0.1:8765 and opens the browser at /docs/index.html. Leave the console window open while browsing; Ctrl+C to stop.

7. CP6606 panel bring-up (TC3 PLC HMI)

The CP6606-0001-0020 is the whole control system in production: one ARM Cortex A8 device under Windows Embedded Compact 7, running both the TwinCAT PLC runtime and a native operator HMI on its 7-inch 800×480 touchscreen. There is no browser and no TcHmiSrv on the panel — the operator UI is Beckhoff's TC3 PLC HMI (visualizations authored inside the PLC project). The web HMI (Flower_HMI) stays on the developer's Windows x64 laptop for engineering use.

Design first, then deploy. Build the six *.TcVISU files in TcXaeShell (right-click PLC1 → Add → Visualization) — see the next card for the per-page design guide. The .TcVISU file format is TcXaeShell-generated XML; hand-authoring is not recommended.

Deployment steps

  1. Add an ADS route to the panel — TcXaeShell → SYSTEM → Routes → Add Route → search the LAN, pick the CP6606, log in with the panel's user (default: no password on Beckhoff CE). Routes must be present on BOTH sides.
  2. Set the target system — the TwinCAT toolbar's target dropdown → pick the CP6606's AMS Net ID. Solution now deploys to the panel instead of localhost.
  3. Enable the SVG→BMP setting — In the Solution Explorer under PLC1, double-click Visualization Manager → tab Settings → check "Automatically convert SVG to BMP". The CE renderer does not support SVG; this is a required setting even if you don't ship any SVG icons today.
  4. Build + Activate Configuration — writes boot files to \Hard Disk\TwinCAT\3.1\Boot\ on the panel. Confirm the PLC comes up in Run mode.
  5. Configure Startup Manager on the panel — on the CE desktop: Start → StartMan → Startup Options tab → Autostart → add an entry with:
    • Type: Application
    • Delay: 30 s (waits for TwinCAT runtime to be Run first)
    • Command: \Hard Disk\TwinCAT\3.1\Components\Plc\Tc3PlcHmi\Tc3PlcHmi.exe
  6. Reboot the panel. After the delay, Tc3PlcHmi.exe should come up fullscreen with your VISU_Home visible. Touch-driven navigation to the subpages should work.
Tc3PlcHmi.ini location has moved. The config-file path changed between TwinCAT builds: C:\TwinCAT\3.1\Components\Plc\Tc3PlcHmi (builds < 4022) → C:\TwinCAT\3.1\Boot\Plc (4022 to 4025) → C:\ProgramData\Beckhoff\TwinCAT\3.1\Boot\Plc\Tc3PlcHmi (4026+). If you ever need to hand-edit the INI, check your panel's TC build first (see the TwinCAT tray → About).

8. PLC HMI design (six VISUs to build in TcXaeShell)

All six visualization pages target 800×480 to match the CP6606 screen. Build them in this order — the shared VISU_Piston tile is referenced by VISU_Pistons, so create it first. Every subpage carries a "← Home" button in the top-left that navigates back to VISU_Home (Element property → Inputconfiguration → mouse down action → Change shown visualization).

  • VISU_Piston — reusable tile, size ~260×220. Interface parameter: piston : ST_HmiPiston. Elements:
    • Label bound to piston.Name (top)
    • Two coloured rectangles for the sensor indicators; color expression → green if piston.bIsExtended/piston.bIsRetracted, grey otherwise
    • Two buttons "EXTEND" and "RETRACT" with Toggle disabled and Tap configured to write piston.bManCmdExtend/bManCmdRetract (TRUE on mouse-down, FALSE on mouse-up — matches the web UC's momentary semantics)
    • Small mode label bound to piston.eMode using a textlist ("Manual" / "Automatic")
  • VISU_Pistons — 800×480. Six Frame elements referencing VISU_Piston in a 3×2 grid; pass GVL_HMI.stSepSol1..stSepSol3, stPushSol1..stPushSol3 to each. Plus one "← Home" button.
  • VISU_Robot — 800×480. Wide banner rectangle at top; color expression on GVL_Robot.stRobot.eConnState (0=grey, 1=yellow, 2=green, 3=red). Text overlay bound to stRobot.sConnStateText. Three "AT POS N" indicator rectangles bound to bAtPos1/2/3. Packet counter (nPacketsRx) and last message (sLastMessage) as text fields. "← Home" button.
  • VISU_AutoCycle — 800×480. Step banner (color expression on GVL_HMI.stSepSol1Auto.eStep, same rules as the web UC). Four buttons Start/Stop/Reset/Ack wired to stSepSol1Auto.bStart/bStop/bReset/bAck. "Continuous" checkbox bound to GVL_HmiPersistent.stSepSol1AutoCfg.bContinuous. Four numeric inputs for tDwellExtendedMs / tDwellRetractedMs / tTimeoutExtendMs / tTimeoutRetractMs. Error text field bound to stSepSol1Auto.sErrorText. "← Home" button.
  • VISU_Diagnostics — 800×480. One ToggleSwitch bound to GVL_Log.bDebugMode. Optional: one text field showing the last log entry (GVL_Log.aLog[(GVL_Log.nWriteIdx - 1) MOD 256].sMsg). "← Home" button.
  • VISU_Home — 800×480. Four large touch buttons in a 2×2 grid: "Pistons" / "Robot" / "Auto" / "Diagnostics", each configured with Inputconfiguration → mouse down → Change shown visualization → target the corresponding VISU. Aim for 400×240 per button so they're easily hit on the 7-inch touchscreen.
What this design deliberately drops vs the web HMI (all acceptable on the panel): no JOG buttons on the piston tile, no localization (English only for v1), no simulation checkboxes (operators shouldn't fake sensors on the panel — that's an engineering task, done via the web HMI or a Watch window). Add them back if the operators ask.

Startup order

After a cold boot, bring the stack up in this order:

Daily startup checklist

  1. TwinCAT runtime → Run mode. Usually automatic on boot. Verify with the TwinCAT icon in the tray (green = Run).
  2. Robot controller. Power it up first — it's the TCP server, so it must be listening at 192.168.201.1:6001 before the bridge tries to dial in. In a bench setup with no real robot, run any TCP server on that port (the bridge GUI's simulator panel does this for you).
  3. HMI server. The published TcHmiSrv runs as a Windows service; confirm http://127.0.0.1:3003 is reachable.
  4. Python bridge. Double-click RobotBridge\start_bridge.bat, or start it as a Scheduled Task / NSSM service. Logs should show: 
    YYYY-MM-DD HH:MM:SS  INFO     loaded config: role=client host=192.168.201.1 port=6001 ams=127.0.0.1.1.1
YYYY-MM-DD HH:MM:SS  INFO     ADS opened to 127.0.0.1.1.1:851
YYYY-MM-DD HH:MM:SS  INFO     connecting to 192.168.201.1:6001
YYYY-MM-DD HH:MM:SS  INFO     connected
  5. Browser. Open the HMI. Go to the Robot page — connection indicator should be yellow (Connecting) briefly, then green (Connected).
  6. Master cycle. Optionally open the AutoCycle page. On operator START (or on the first POS1 from the robot), the FB emits AUTO_STARTED, forces the enabled pistons into Automatic, and drives through the ten-state sequence.

What each state means on the HMI Robot page

StateColorMeaning
DisconnectedGreyBridge is in the reconnect-delay sleep between dial attempts, or has just started and hasn't dialed yet.
ConnectingYellowBridge is calling connect() against 192.168.201.1:6001. Usually flashes by if the robot is reachable.
ConnectedGreenSocket up. Bridge is receiving frames and polling Tx flags.
ErrorRedBridge caught an exception (ADS write failed, socket error, bad config). Check the bridge console for the traceback.
State can go stale. If the bridge process dies, the HMI keeps showing the last-written state indefinitely — v1 has no PLC-side watchdog. If Connected is shown but nPacketsRx hasn't moved in minutes, check the bridge is alive.

Unattended startup (auto-run the bridge)

Two options:

  • Windows Scheduled Task. Task Scheduler → Create Basic Task → Trigger "At log on" → Action "Start a program" → RobotBridge\start_bridge.bat. Set "Run whether user is logged on or not" so it survives RDP disconnects.
  • NSSM service. nssm install FlowerRobotBridge, point at python.exe with args robot_bridge.py --config config.yaml, set the working directory to RobotBridge\. Survives reboots and runs headlessly.

Smoke test — robot ↔ bridge ↔ PLC ↔ HMI

End-to-end verification without a real robot. In production the robot is the TCP server and the bridge is the client; for the smoke test we flip that around and let the GUI's simulator play a client against a server-mode bridge. Two options:

  • GUI (recommended) — RobotBridge\start_gui.bat. One window with three panels — bridge process, PLC live state, and a robot-simulator client that opens a TCP connection to the bridge and lets you send POS / RESET_ERROR frames button-by-button. Every send shows the exact bytes on the wire.
  • Manual PowerShell TCP server — keeps the shipped client-mode bridge unchanged; the PowerShell script hosts a listener on :6001 that the bridge dials into. Good for CI or headless boxes. See the "Manual" card below.

Option A — the GUI (bridge_gui.py) — full walk-through

A single-window tkinter desktop app that ties three things together — no extra dependencies beyond what the bridge already needs (pyads, pyyaml).

  • Bridge process panel — Start / Stop robot_bridge.py as a subprocess. Live stdout tail. Bridge PID + colored LED.
  • PLC live state panel — reads GVL_Robot.stRobot via pyads 5× per second and shows: connection state (colored banner), three AT POS N LEDs, packet counters (Rx + Tx), last messages (Rx + Tx).
  • Robot simulator panel — opens a TCP client to the bridge and sends POS1 / POS2 / POS3 / RESET_ERROR / custom-frame at the click of a button. Every send shows the exact bytes that left the process, hex + ASCII, so you can see the framing (each frame ends in 0x0A = \n).
  • Event log — merged feed of GUI events + bridge stdout + PLC state changes. Colored per source.
  • "What is happening right now" panel — describes the current step in plain English: what "listening" means, what a client does, why the position bits latch, what an ADS write does.
Flip role before running. The GUI's simulator is a TCP client. To make it work, temporarily edit config.yaml: 
robot:
  role: "server"     # was "client"
  port: 6001
When the walkthrough is done, revert to role: "client". The shipped default matches production (robot is the server), but the GUI's tutorial-friendly path is the mirror direction.

Launch:

python RobotBridge/bridge_gui.py --config RobotBridge/config.yaml

# or double-click on Windows:
RobotBridge\start_gui.bat

Recommended step-by-step:

  1. Bridge panel → Start bridge. Log shows listening on :6001. PLC live state shows eConnState = Connecting (yellow).
  2. Simulator panel → Connect to 127.0.0.1:6001. The panel goes to "connected"; PLC panel flips to Connected (green).
  3. Simulator → POS1. Byte view shows 50 4F 53 31 0A. PLC panel: bAtPos1 = TRUE, nPacketsRx = 1, sLastMessage = "POS1". Master AutoCycle can now trigger on Pos1 rising if it's IDLE.
  4. Simulator → POS2. Same, but bAtPos2 latches (POS1 clears). If the master is in WAIT_POS2, it advances to PUSH_EXTENDING.
  5. AutoCycle page in browser → START. Bridge log shows outbound send: tx: 'AUTO_STARTED'. Simulator panel doesn't currently display inbound frames from the bridge, but PLC panel's nPacketsTx increments and sLastTxMessage updates.
  6. Complete the cycle either by pressing the per-step Sim checkboxes on the AutoCycle page (fastest) or by ticking each piston's Simulate ✓ Extended/Retracted to satisfy the sensor AND. Watch tx: 'PUSH_DONE' land on PUSH_RETRACTING complete.
  7. Force an error: with no sims and no real sensors, wait for the step timeout. Bridge log shows tx: 'PISTONS_ERROR', PLC iErrorCode non-zero.
  8. Simulator → RESET_ERROR. Byte view shows the frame. PLC bRxResetError pulses; FB consumes it and drops back to IDLE.

Option B — manual PowerShell TCP server

Same underlying protocol, no GUI. Uses only .NET stdlib. Run this before the bridge so the bridge has something to dial into.

$listener = [System.Net.Sockets.TcpListener]::new([System.Net.IPAddress]::Any, 6001)
$listener.Start()
Write-Host "listening on :6001"
$client = $listener.AcceptTcpClient()
Write-Host "bridge connected"
$stream = $client.GetStream()
$w = [System.IO.StreamWriter]::new($stream)
$w.NewLine = "`n"; $w.AutoFlush = $true

$w.WriteLine("POS1"); Start-Sleep 1
$w.WriteLine("POS2"); Start-Sleep 1
$w.WriteLine("POS3"); Start-Sleep 1
$w.WriteLine("RESET_ERROR")
Start-Sleep 3
$client.Close(); $listener.Stop()

Point config.yaml → robot.host at 127.0.0.1 for a local run, or the machine's LAN IP for a two-box test.

Prep

  1. PLC in Run mode with the latest build activated.
  2. HMI running; browser open on the Robot page.
  3. Something acting as the robot's TCP server on :6001 — either the GUI's simulator panel or the PowerShell one-liner above.
  4. Bridge running in client mode (robot.role: "client", host: "192.168.201.1" or wherever your server is, port: 6001). Start via start_bridge.bat or python robot_bridge.py --config config.yaml.
  5. Bridge console shows connected.
  6. HMI Robot page shows: eConnState = Connected (green), all AT POS N grey.

Expected HMI behavior

  • Robot page's connection indicator flips green (Connected) as soon as the bridge dials in.
  • After POS1: AT POS 1 lights green. nPacketsRx = 1. sLastMessage = "POS1".
  • After POS2: POS 1 clears, POS 2 lights. nPacketsRx = 2.
  • After POS3: POS 2 clears, POS 3 lights. nPacketsRx = 3.
  • Master cycle START on the AutoCycle page: nPacketsTx = 1, sLastTxMessage = "AUTO_STARTED".
  • After the simulated cycle completes: sLastTxMessage = "PUSH_DONE", nCyclesCompleted increments.
  • Bridge disconnect scenario: stop the simulator's TCP server. Bridge logs the drop; HMI eConnState → Disconnected (grey). reconnect.delay_seconds later, bridge redials. If the server is back up: green again. If not: it cycles Connecting → Error → Disconnected → Connecting until it succeeds.

Verify from Python (no HMI needed)

If the HMI isn't up yet, you can still confirm the whole chain by reading GVL_Robot.stRobot directly via pyads. Handy for CI / smoke-test scripts:

python -c "
import pyads
plc = pyads.Connection('127.0.0.1.1.1', 851); plc.open()
for f, t in [('bAtPos1', pyads.PLCTYPE_BOOL), ('bAtPos2', pyads.PLCTYPE_BOOL),
             ('bAtPos3', pyads.PLCTYPE_BOOL), ('nPacketsRx', pyads.PLCTYPE_UDINT),
             ('nPacketsTx', pyads.PLCTYPE_UDINT),
             ('sLastMessage', pyads.PLCTYPE_STRING),
             ('sLastTxMessage', pyads.PLCTYPE_STRING)]:
    print(f, '=', plc.read_by_name(f'GVL_Robot.stRobot.{f}', t))
plc.close()
"

Piston smoke test (independent of the robot)

Even without the bridge running, the pneumatic pistons are fully operable from the HMI:

  1. Go to the Piston page. Six UCs in a 3×2 grid.
  2. For SepSol1 or PushSol1 (wired to real IO), press EXTEND. Solenoid indicator turns green, coil energizes and stays energized (SOL-Spring latch). Retract by pressing RETRACT — coil drops, spring returns the piston.
  3. Press-and-hold JOG next to EXTEND — coil energizes only while the button is held; releasing lets the spring pull the piston back.
  4. For SepSol2/3 + PushSol2/3 (soft coil, no DI yet), the coil write reaches dOut[2..3] and dOut[5..6] — the DI sensor indicators just don't light up until real switches are wired. See Open issues.

Recovery

What to do when something has stopped working.

Symptom: HMI Robot page frozen on stale state

Typically means the bridge process died. Position indicators + connection state reflect the last successful write.

  1. Check the bridge console window — is it closed / exited?
  2. Restart via start_bridge.bat. Watch for exceptions on startup.
  3. If the exception is ADSError: 1861 (target port not found) — the TwinCAT runtime is not in Run mode. Restart it.
  4. If ConnectionRefusedError in client mode — the robot's TCP server is down or unreachable.

Symptom: HMI shows HMI_ADS_E_INVALID_SYMBOL for stRobot fields

The PLC symbol table doesn't yet know about the new struct fields. Fix:

  1. In TcXaeShell: PLC → Login, then TwinCAT → Activate Configuration. An Online Change is not sufficient for new struct members — the symbol table cache needs a full activate.
  2. If still failing after Activate, restart TcXaeShell — the IDE has a symbol-table cache of its own.
  3. Force-refresh the browser (Ctrl+F5) to bypass the HMI's client cache.

Symptom: SYMBOL_NOT_AUTHORIZED on stRobot

The symbol is in the SYMBOLS map but not in the USERGROUPS ACL. Fix:

  1. Open Server/TcHmiSrv/TcHmiSrv.Config.default.json.
  2. Under USERGROUPS.__SystemUsers.SYMBOLS, verify "ADS.PLC1.GVL_Robot.stRobot": 3 is present.
  3. Rebuild + republish the HMI project.

Symptom: HMI page shows old content after edit

TcHmi caches .content and .usercontrol in the browser aggressively.

  1. Hard reload: Ctrl+F5, or open DevTools and check "Disable cache" while devtools is open.
  2. If a UserControl was renamed, also check tchmiconfig.json's userControls list.
  3. If a localization key was added, both en.localization and de.localization must have it — otherwise the runtime falls back to the literal key text.

Symptom: PLC won't build after external file edit

Common when edits happen outside TcXaeShell (e.g. via Claude Code or git).

  1. Confirm every new file has a matching <Compile Include="..."/> in PLC1/PLC1.plcproj.
  2. Confirm every TcPOU / TcDUT / TcGVL has a unique GUID in its Id="{...}" attribute.
  3. If the project appears empty in the Solution Explorer after opening, look for a stripped .tsproj — never commit hand-edited .tsproj files. Restore from git if needed.

Symptom: HMI console shows E_TRIGGER_CONDITION_INVALID (code 5200/0x1450)

Full text: "Condition definition is invalid because it contains no expressions and no following actions." The event target usually points at a specific property binding, e.g. %pp%RobotHost::robot::eConnState%/pp%.

Cause. A trigger's Condition action was authored with an elseIf part. TcHmi only supports if and else. The elseIf key is silently ignored — the part is seen as having no expression and no action, and the whole condition is refused at parse time.

Fix. Nest a fresh Condition inside the outer else's then array, one level per additional branch:

parts: [
  { if: [state == A], then: [action A] },
  { else: [
      { objectType: "Condition", parts: [
          { if: [state == B], then: [action B] },
          { else: [
              { objectType: "Condition", parts: [
                  { if: [state == C], then: [action C] },
                  { else: [action D] } ] } ] } ] } ] }
]

Verbose but the only supported shape for n-way state → color mapping without introducing a user function. The Flower UC_Robot.usercontrol uses this nesting for its 4-state eConnState color trigger — copy that pattern.

Symptom: CP6606 boots to CE desktop, no HMI shown

The Beckhoff Startup Manager didn't fire the PLC HMI Client, or the PLC isn't in Run mode when the client starts.

  1. On the panel, tap Start → StartMan → verify the Autostart entry exists with the correct path \Hard Disk\TwinCAT\3.1\Components\Plc\Tc3PlcHmi\Tc3PlcHmi.exe.
  2. Increase the delay to 45–60 s if the PLC needs longer to reach Run mode.
  3. Manually double-click Tc3PlcHmi.exe from the CE file explorer. If it opens a "no visualization found" or "cannot connect to PLC" message, the PLC isn't running or the config INI is wrong.
  4. From TcXaeShell (on the developer laptop, connected via ADS route), check the CP6606's PLC state — it must be Run for the client to have anything to render.

Symptom: CP6606 HMI shows blank / missing images

The CE renderer requires BMP; SVG isn't rendered.

  1. In TcXaeShell → PLC1 → Visualization Manager → Settings tab → check "Automatically convert SVG to BMP".
  2. Rebuild + Activate. The panel needs a fresh boot-file deployment.
  3. If images still don't appear, verify the source files are actually in the project (right-click Image Pool → Properties) — missing image references silently render as blanks.

Symptom: docs viewer shows a fetch error

viewer.html uses fetch() to load the .md source, which browsers block under file://.

  1. Serve via scripts\serve-docs.bat (or python -m http.server 8765 from the repo root) and browse via http://127.0.0.1:8765/docs/index.html.
  2. The HTML pages (configuration, operation, open-issues) work fine under file://; only the markdown viewer needs HTTP.
